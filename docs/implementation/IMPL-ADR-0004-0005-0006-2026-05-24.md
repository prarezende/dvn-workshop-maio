## Summary

Implemented three interdependent ADRs covering the complete GitOps CI/CD chain:

1. **ADR-0004**: New Terraform stack `03-ci-cd-stack-ai` with OIDC provider for GitHub Actions federation and IAM role with least-privilege ECR push permissions.
2. **ADR-0006**: ArgoCD Application CRD manifest at `dvn-workshop-kubernetes/argocd-application.yaml` for GitOps delivery to EKS.
3. **ADR-0005**: GitHub Actions workflow at `.github/workflows/ci-cd.yml` with path-filtered builds for backend and frontend, and automatic kustomization tag update.

## Architectural Decisions Made During Implementation

### 1. `thumbprint_list` omitted from `aws_iam_openid_connect_provider`
Per Terraform MCP validation of doc ID 12310858 and the ADR explicitly calling this out: AWS uses its own trusted CA library for GitHub's OIDC endpoint. `thumbprint_list` was omitted, matching the ADR's instruction.

### 2. Trust policy uses dynamic `sub` condition via `for` expression
The `allowed_branches` list in the variable is converted to a list of `repo:<org>/<repo>:ref:refs/heads/<branch>` strings using a `for` expression in `locals`. This allows adding branches without changing HCL code — only the tfvars need updating.

### 3. `prevent_destroy = true` on OIDC provider
The ADR explicitly calls for this lifecycle protection. Accidental deletion would break all pipelines. Applied to `aws_iam_openid_connect_provider.github`.

### 4. `aws_iam_policy_document` data sources used instead of inline JSON
Preferred over raw JSON strings as it produces validated, properly-escaped HCL and is more readable. Both the trust policy and the ECR push policy use this pattern.

### 5. ArgoCD Application CRD NOT added to `kustomization.yaml`
The `argocd-application.yaml` is intentionally NOT included in the kustomization's `resources` list. If ArgoCD managed its own Application CRD via the same kustomization it was syncing, a deletion of the Application resource would trigger ArgoCD to delete and recreate itself. The Application CRD is a one-time bootstrap resource applied manually once via `kubectl apply`.

### 6. `build-backend` and `build-frontend` as separate jobs (not matrix)
The ADR allowed matrix strategy but the separate-job approach was chosen for clarity and because each job has a distinct `if` condition based on `detect-changes` outputs. This avoids matrix complexity with heterogeneous conditions and produces cleaner logs per application.

### 7. GitHub Actions workflow `update-kustomization` condition
The `if: always() && ...` pattern ensures the job runs even when one of the build jobs is skipped (because only one app changed). The condition checks that at least one build succeeded and neither failed.

### 8. `vars.AWS_ROLE_ARN` (variable, not secret)
The role ARN is not a secret — it is a public IAM resource identifier. Using `vars.` (GitHub Actions variable) instead of `secrets.` is the correct pattern. The human must create this variable in the GitHub repository settings after deploying ADR-0004.

## Post-Deploy Validation Steps Required

### After `terraform apply` of `03-ci-cd-stack-ai`:
```bash
# Verify OIDC provider created
aws iam list-open-id-connect-providers

# Verify role exists and has correct trust policy
aws iam get-role --role-name dvn-workshop-github-actions-role

# Verify policy attached
aws iam list-attached-role-policies --role-name dvn-workshop-github-actions-role

# Get the role ARN for the GitHub Actions variable
aws iam get-role --role-name dvn-workshop-github-actions-role \
  --query 'Role.Arn' --output text
```

### After ArgoCD installation (manual bootstrap):
```bash
# 1. Configure kubectl
aws eks update-kubeconfig --name dvn-workshop-production --region us-east-1

# 2. Install ArgoCD
kubectl create namespace argocd
kubectl apply -n argocd -f \
  https://raw.githubusercontent.com/argoproj/argo-cd/v2.14.11/manifests/install.yaml

# 3. Wait for pods to be ready
kubectl wait --for=condition=Ready pods --all -n argocd --timeout=300s

# 4. Apply the Application CRD (bootstrap step)
kubectl apply -f dvn-workshop-kubernetes/argocd-application.yaml

# 5. Get initial admin password
kubectl -n argocd get secret argocd-initial-admin-secret \
  -o jsonpath="{.data.password}" | base64 -d

# 6. Access the UI via port-forward
kubectl port-forward svc/argocd-server -n argocd 8080:443
```

### After setting up GitHub Actions:
```bash
# Set the AWS_ROLE_ARN variable in GitHub repository settings
# Settings -> Secrets and variables -> Actions -> Variables -> New repository variable
# Name: AWS_ROLE_ARN
# Value: arn:aws:iam::654654554686:role/dvn-workshop-github-actions-role

# Trigger a pipeline by pushing a change to dvn-workshop-apps/
# Verify workflow run
gh run list --workflow=ci-cd.yml

# Verify image in ECR
aws ecr describe-images \
  --repository-name dvn-workshop/production/backend \
  --region us-east-1

# Verify kustomization commit
git log --oneline --grep="ci: update image tags"
```

## Deviations from ADR

| Item | ADR Spec | Implemented | Justification |
|------|----------|-------------|---------------|
| Trust policy `sub` condition | Hardcoded string for `main` branch | Dynamic list from `allowed_branches` variable | More maintainable; same security posture; easily extensible |
| Build jobs | Matrix strategy suggested | Separate conditional jobs | Cleaner conditional logic; avoids matrix heterogeneity |
| `thumbprint_list` | Omit | Omitted | Confirmed via MCP — AWS handles GitHub CA automatically |
| ArgoCD Application in kustomization | Not specified | NOT included | Avoid self-referential management loop |

## Rollback Plan

### ADR-0004 (Terraform IAM/OIDC):
```bash
# Stack has prevent_destroy on OIDC provider.
# To destroy: remove lifecycle block first, then:
cd dvn-workshop-terraform/03-ci-cd-stack-ai
terraform destroy -var-file="envs/production.tfvars"
# No data loss — resources are IAM only.
```

### ADR-0006 (ArgoCD):
```bash
# Delete the Application CRD (stops ArgoCD from managing workloads):
kubectl delete application dvn-workshop -n argocd

# Uninstall ArgoCD:
kubectl delete namespace argocd

# Workloads in 'default' namespace remain untouched after deletion.
```

### ADR-0005 (GitHub Actions workflow):
```bash
# Revert the kustomization commit to roll back a bad deploy:
git revert <commit-sha>
git push origin main
# ArgoCD detects the revert and rolls back the deployment automatically.
```
